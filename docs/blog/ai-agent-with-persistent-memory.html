How to build an AI agent with persistent memory

Updated April 2026 · Lucas Topinka

Most AI agents forget everything the moment the conversation ends. If you want an agent that remembers you across sessions — projects, preferences, facts, rules — you need a proper memory system. This post walks through the four-layer architecture I built for NEXUS and the trade-offs I hit along the way.

Why "chat history" isn't enough

The naive approach is to store every message and stuff them all into the next prompt. That breaks quickly:

  • Context window fills up. Even 200K tokens runs out after a few weeks of active use.
  • Signal drowns in noise. The model sees every side comment and typo alongside the durable facts that actually matter.
  • No retrieval strategy. If you ask "what did we decide about X?", the model has to scan the whole transcript.

A real memory system needs different storage strategies for different kinds of knowledge, and a way to consolidate raw experience into durable, retrievable facts.

Four-layer memory

NEXUS splits memory into four layers:

1. Episodic memory

Every conversation, as it happened. Raw, unfiltered. This is the source material.

2. Semantic memory

Extracted facts about the user, their projects, their world. "Lucas is a developer building NEXUS. Uses Claude. Prefers concise replies." This is what gets injected into future prompts.

3. Procedural memory

Learned rules and patterns. "Always run tests after code changes. Never mention OpenClaw in the NEXUS repo." Behavioural guidance.

4. Short-term / working memory

A rolling window of the last ~50 messages, always in context. This is the conversational near-field.

The SQLite schema

One table does it all. Each row has a layer and a type, plus metadata for retrieval:

CREATE TABLE memories (
  id              TEXT PRIMARY KEY,
  layer           TEXT NOT NULL CHECK(layer IN ('buffer','episodic','semantic','procedural')),
  type            TEXT NOT NULL CHECK(type IN (
    'conversation','task','fact','preference','workflow',
    'contact','opinion','mistake','procedure'
  )),
  content         TEXT NOT NULL,
  summary         TEXT,
  importance      REAL NOT NULL DEFAULT 0.5,
  confidence      REAL NOT NULL DEFAULT 1.0,
  emotional_valence REAL,
  created_at      TEXT NOT NULL DEFAULT (datetime('now')),
  last_accessed   TEXT NOT NULL DEFAULT (datetime('now')),
  access_count    INTEGER NOT NULL DEFAULT 0,
  tags            TEXT NOT NULL DEFAULT '[]',
  related_memories TEXT NOT NULL DEFAULT '[]',
  source          TEXT NOT NULL DEFAULT 'system',
  metadata        TEXT NOT NULL DEFAULT '{}'
);

CREATE INDEX idx_memories_layer ON memories(layer);
CREATE INDEX idx_memories_type ON memories(type);
CREATE INDEX idx_memories_importance ON memories(importance DESC);

Key fields:

  • importance (0..1) — how much to weight during retrieval. Set at creation, updated during consolidation.
  • confidence (0..1) — how sure the system is that this is true. Evolves with contradicting / reinforcing evidence.
  • last_accessed + access_count — for LRU-style decay.
  • tags + source — for filtered retrieval and audit.

The dream cycle (consolidation)

Raw episodic memory doesn't help future prompts directly — there's too much of it. You need a consolidation step that promotes durable insights into the semantic layer.

NEXUS runs a nightly dream cycle between 2am and 5am local time. It:

  1. Reads the day's episodic entries
  2. Clusters related ones
  3. Extracts durable facts → writes to semantic layer with an importance score
  4. Detects contradictions with prior memories → flags for review
  5. Applies decay to old low-importance memories
  6. Generates reflections ("what patterns did I see today?") + ideas ("what should we do tomorrow?")

This mirrors how human sleep consolidates short-term memory into long-term. Do it at night. I originally ran it on a 6-hour interval and users got random "insight" messages during the workday — bad UX. Time-of-day gating solved it.

Retrieval strategy

When a new message comes in, NEXUS builds context by layer:

  1. Short-term: the last ~50 messages, always included
  2. Semantic: top 20 by embedding similarity to the current message
  3. Procedural: all active rules, filtered by tags relevant to the current activity
  4. Episodic: nothing by default — these are raw material, only touched by the consolidation job

Plus a time capsule subsystem that surfaces aged (14+ day) high-importance memories when the current question is similar to one from the past.

Model tiering for cost + latency

Using Opus for everything is expensive. Using Haiku for everything is dumb. NEXUS routes sub-tasks to the right model:

  • Opus 4.7 — task planning, Ultra-mode review, Code Dreams meta-review. High-stakes, low-frequency.
  • Sonnet 4.6 — chat, execution, memory synthesis, vision. The workhorse.
  • Haiku 4.5 — readiness checks, summaries, fallbacks. Cheap and fast.

Memory consolidation uses Sonnet. Post-response self-evaluation uses Haiku. This three-tier routing cut my daily cost from ~$15 (Opus-only) to ~$2 without losing quality where it matters.

LLM-driven memory import

When a new user installs NEXUS, they've probably accumulated context in other AI tools (Claude Code, Codex, Gemini CLI, Cursor). Rather than ignore that, NEXUS hands the raw content to Sonnet with a synthesis prompt:

You are NEXUS — a personal AI agent that lives on a user's Mac.
You are about to import context from another AI assistant.
Your job: read the raw content and write YOUR OWN memories and skills
IN YOUR OWN VOICE. Do not copy verbatim. Synthesize. Distill patterns.

Output JSON: { memories: [...], skills: [...] }

On my own machine, 41 Claude Code notes + 198 Codex rules → 26 distilled memories + 7 auto-generated skill files. The LLM writes in first-person NEXUS voice, extracting things like:

  • User facts — who they are, what they do
  • Preferences — how they like replies formatted
  • Workflow habits — tools they use, commands they run
  • Explicit rules — things they've asked for or strict no-gos
  • Skills — repeatable procedures worth codifying

A deterministic fallback extractor runs if the LLM call fails, so users always get something imported.

What I'd do differently

  • Add embeddings earlier. NEXUS ships with optional semantic embeddings (OpenAI or local). Turning them on from day 1 would improve semantic retrieval quality.
  • Confidence decay should be smarter. Currently it's time-based. A better approach: decay faster on contradicting evidence, slower on reinforcement.
  • Separate read/write connections. Better-sqlite3 is synchronous; on write-heavy loads this can bottleneck. Not a problem at my usage level yet.
